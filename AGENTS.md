# AGENTS

このリポジトリでAIに作業を依頼するときの、進め方のルール。

## ルール

### 許可なく編集しない

- ユーザーが「コードを書いて/実装して/変更して」など、明確に実装を依頼していない限り、勝手にファイルを編集しない（`apply_patch` を含む）。
- 変更や実行が必要そうなときは、先に次を短く提示して確認を取る。
  - 何をするか（対象ファイル、実行コマンド）
  - なぜ必要か（目的）
  - 影響（差分の大きさ、生成物、環境要件など）

### コード変更開始前に差分確認する

- コード変更（`apply_patch` を含む）を始める前に、原則 `git status --porcelain` と必要に応じて `git diff` で作業ツリーの差分を確認する。
- そこで見つかった「元から存在していた差分」は原則タスク外として無視し、勝手に直さない。
  - ただし、変更対象ファイルと差分が重なっていて作業に影響する場合は、その旨をユーザーに共有し、どう扱うか確認してから進める。

#### 「依頼外の整備変更」は必ず確認する

タスク達成に直接必要でないが、品質維持のために有益な変更（いわゆる hygiene / cleanup）は、AIの判断だけで混ぜない。
必ず事前に「入れてよいか」確認してから行う。

例:

- `.gitignore` の追加・変更
- 生成物（`output/` 配下など）の削除
- README の追記・修正（仕様説明の同期を含む）
- フォーマッタ/リンタの一括適用
- 依存関係のアップデート（`npm install` の結果更新など）

### devcontainerを優先して追記する

- OSパッケージ追加やツール導入など「環境変更」が必要になったら、手動手順の案内だけで済ませず、原則として `.devcontainer/` に反映する方針を優先する。
- 例: `apt-get install` が必要なら、[.devcontainer/Dockerfile](.devcontainer/Dockerfile) の `apt-get install` に追加して再現性を担保する。
- ただし、リポジトリの意図（最小構成/サイズ/ビルド時間）に反しそうな場合は、追加前に必ず相談する。

#### 環境変更の確認テンプレ（短く）

環境変更（apt追加、ツール導入、Dockerfile変更など）を行う前に、次を1〜2行で提示して合意を取る。

- 何を入れるか（パッケージ/ツール名、追記先）
- なぜ必要か（目的）
- 影響（イメージサイズ/ビルド時間、再ビルド要否、既存環境への影響）

例: 「LLVM用に `.devcontainer/Dockerfile` に `clang/llvm/lld` を追加します。イメージが大きくなり、Rebuildが必要になりますがよいですか？」

### 生成物（build artifacts）の扱い

- 生成物（例: `output/*.ll`, `output/*` の実行ファイルなど）は原則コミットしない。
- `output/` のようにディレクトリだけ欲しい場合は `.gitkeep` で維持する。
- 生成物の削除（`rm`）や `.gitignore` 調整は「依頼外の整備変更」に該当するため、事前確認が必要。

### 実行コマンドの確認ライン

ユーザーが明確に実装を依頼している場合でも、次のような副作用が大きい操作は、実行前に一言確認する。

- `sudo apt-get ...` などの環境変更
- `rm`, `mv`, 大量のファイル変更/削除
- 長時間・大量出力になりうるコマンド（広範なテスト、ビルド、フォーマット全適用など）

## Tips

#### シェル履歴展開（`!`）に注意

bash等の対話シェルでは `!` が「履歴展開」として解釈され、`event not found` になることがある。
コマンド例を提示/実行するとき、`!` を含む可能性がある場合は次のいずれかで回避する。

- `!` を含む部分をシングルクォートで囲む（例: `echo 'a!b'`）
- `!` をエスケープする（例: `echo a\!b`）
- 一時的に履歴展開を無効化する（例: `set +H` または `set +o histexpand`）
